# 야간절전 아침 복원 점검

야간절전(매일 02:00~10:00 KST) 후 기반·애드온이 정상 복원됐는지 점검한다.
아침 복원 체크리스트 **1단계(배포팀 담당)** 를 자동화한 것이다.

## 사용법

```bash
./scripts/morning/verify.sh
```

10:10경 한 번 실행하고, 통과하면 앱팀에 "테스트 가능" 알림을 보낸다.

| 종료 코드 | 의미 | 조치 |
| --- | --- | --- |
| 0 | 전부 통과(또는 절전 시간대라 생략) | 앱팀에 알림 |
| 1 | WARN 있음 | 몇 분 뒤 재실행. 출력된 조치 명령 확인 |
| 2 | FAIL 있음 | 배포팀 → 이미선님(인프라) → 팀장 순으로 공유 |

## 점검 항목

| # | 항목 | 근거 |
| --- | --- | --- |
| 1 | 컨텍스트가 `hailcast-dev-eks` 인지 | 틀린 클러스터를 점검하면 나머지가 전부 무의미 |
| 2 | Ready 노드 2대 | 규약서 §5-8, 10:00 노드그룹 min2/desired2 |
| 3 | RDS `available` | 규약서 §5-8, 09:50 시작(기동에 수 분) |
| 4 | argocd·external-secrets 파드 | ArgoCD → ESO 순으로 복원 |
| 5 | `hailcast-rds-secret` 재생성 | 밤새 사라졌다가 ESO가 다시 만듦 |
| 6 | 앱 파드 기동 | Secret이 앱보다 늦으면 CrashLoop/ConfigError |
| 7 | 전체 Application Synced/Healthy | 위가 정상이어도 sync가 틀어질 수 있음 |

## 설계상 알아둘 것

**절전 시간대엔 점검하지 않는다.** 02:00~10:00 사이에 실행하면 "정상적으로 절전 중"이라고
알리고 종료한다. 이 시간대의 노드 0대·파드 없음은 정상이며 장애가 아니다.

**복원 직후(10:00~10:15)는 관대하게 판정한다.** 이 구간의 미완성은 FAIL 대신 WARN이다.
노드가 NotReady이거나 RDS가 아직 기동 중인 게 이 시각엔 정상이기 때문이다.

**고치지 않고 알려주기만 한다.** 문제를 발견해도 파드를 지우거나 sync를 걸지 않는다.
실행할 명령을 출력만 하고, 판단과 실행은 사람이 한다.

**클러스터에서만 고친 건 되돌아간다.** 출력된 조치 명령으로 임시 복구를 했더라도, 설정
변경이라면 git에도 반영해야 한다. selfHeal이 다음 sync에서 git 기준으로 되돌린다.

## 아직 안 한 것

자동 실행(스케줄러)과 결과 알림은 넣지 않았다. 아침 복원 실패는 "클러스터가 안 올라온
상황"이라 클러스터 안의 CronJob으로 감시하면 감시자도 같이 죽는다. GitHub Actions
스케줄처럼 클러스터 밖에서 도는 방식이 맞고, EKS 접근 자격증명(OIDC) 설계가 필요해
별도 과제로 둔다.
